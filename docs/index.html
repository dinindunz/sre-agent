Cache Behavior

Each cycle: BeforeModelCall → Model Call → AfterModelCall

Each completed cycle includes both the user input AND the assistant's response.

After Cycle 1 completes: Full context: [System Prompt + Tools + User1 + Asst1] Cache Read: 0, Cache Write: 0 No caching yet - Bedrock waits for meaningful prefix After Cycle 2 completes: Full context: [System Prompt + Tools + User1 + Asst1 + User2 + Asst2] Cache Read: 0, Cache Write: [System Prompt + Tools + User1 + Asst1] Writing entire Cycle 1 context to cache After Cycle 3 completes: Full context: [... + User2 + Asst2 + User3 + Asst3] Cache Read: [System + Tools + User1 + Asst1] (cached prefix) Cache Write: [User2 + Asst2] (new exchange only) After Cycle 4 completes: Cache Read: [... User1 + Asst1 + User2 + Asst2] (growing prefix) Cache Write: [User3 + Asst3] (new exchange only) After Cycle 5 completes: Cache Read: [... + User3 + Asst3] (growing prefix) Cache Write: [User4 + Asst4] (new exchange only)

How it works: Bedrock waits until there's a meaningful conversation prefix before caching (typically after first exchange). Once cached, the prefix is reused at 90% discount. New turns get incrementally added to the cache for future requests.

Assumptions & Sources
AgentCore pricing: aws.amazon.com/bedrock/agentcore/pricing — Bedrock model pricing: aws.amazon.com/bedrock/pricing (Asia Pacific Sydney, April 2026).
All prices from published AWS rate cards (USD), converted to AUD at the rate defined in config.js. AgentCore Runtime uses active resource consumption billing — you only pay for actual CPU usage; I/O wait (LLM responses, tool/API calls, DB queries) incurs no CPU charge. Memory billed on peak consumption per second (128 MB minimum). Billing spans microVM boot through shutdown, including system overhead. Storage (ECR/S3) and network data transfer billed separately at standard rates.
